## 내가 하려고하는 것
- 회장님이 개발 아이디어를 주심.
- 나의 역할은 그것을 구체화하는 것이다. 구체화는 아래 항목을 포함한다.
	- PRD(제품 요구사항 문서)를 작성하기.
	- 부품의 크기 및 재료 등을 결정하기 — 인간(엔지니어)이 판단하고, LLM이 기록·정리함.
	- 관련 설계기준(규격, 표준)을 찾아 정리하기 — 인간은 검색·선정, LLM은 요약·인용 정리.
	- 구조계산 방법과 절차 수립(손계산 및 유한요소 해석 워크플로우) — 인간이 방식 설정, LLM이 문서화.
	- 실제 구조계산 수행(수치해석, Abaqus 등 유한요소 해석) — 인간이 수행.
	- 계산 결과에 따른 분기 결정:
		- 시나리오 A: 계산 결과가 기준을 만족하지 못함 → 변수 재검토, 설계조건 수정, 대안 제시(반복).
		- 시나리오 B: 계산 결과가 만족 → 상세도면·제작성 검토로 진행.
	- 제작 가능성 및 경제성 검토(도면화, 공급사·업체 연락, 견적 확인).
	- 필요 시 실험계획 수립 및 시험 수행.

### 목적
- 회장님이 제시한 아이디어를 엔지니어링 관점에서 실현 가능한 제품·프로젝트로 전환하는 것.
- 산출 가능한 결과물(문서·계산·도면·시험계획)을 만들고 의사결정 포인트를 명확히 한다.

### 산출물(예시)
- `PRD.md` : 목표, 성능지표, 사용환경, 제약조건, 승인 기준 등.
- `Progress.md` : 현재 단계, 완료 항목, 다음 작업, 리스크, 의사결정 포인트.
- 계산보고서 : 가정, 해석 방법, 결과, 감도 분석.
- 도면(제작도) 및 제조 관련 코레스폰던스(공급사 이메일 등).
- 실험계획서(필요 시).

### 상세 워크플로우(권장)
- 1) 아이디어 수집 및 목표 정의
	- 회장님과 확인 회의로 요구사항(핵심 성능, 제약)을 정리.
	- 초기 PRD(간단 버전)를 작성.
- 2) 설계 기준·참고문헌 조사
	- 관련 규격·유사제품 조사, 주요 참고자료를 LLM에 요약하게 함.
	- 중요한 인용은 원문 링크/출처 기록.
- 3) 주요 부품·재료 선정 및 초기 치수 산정
	- 인간이 초기 값(치수, 재료)을 정하고 LLM은 정리·버전관리.
- 4) 구조 해석(손계산 → 수치해석)
	- 손계산으로 주요 하중·경계조건 검증.
	- 유한요소 모델링(Abaqus 등)으로 상세 검증 및 감도분석.
- 5) 결과 검토 및 분기
	- 실패 시: 설계 변수 수정·대안 검토(되돌아가기).
	- 통과 시: 제작성·비용검토로 진행.
- 6) 제작성·비용·업체 검토
	- 도면 작성, 업체(예: 센코어테크) 문의 및 비용 확인.
- 7) 시험(필요 시) 및 양산 준비
	- 실험 수행 → 결과 반영 → 최종 도면 확정.

### 문서 템플릿(간단)
- `PRD.md` (권장 항목)
	- 프로젝트 개요
	- 주요 기능/성능 지표(KPI)
	- 사용 환경 및 제약조건
	- 성공 기준(검증 방법)
	- 주요 리스크 및 대응 방안
- `Progress.md` (권장 항목)
	- 현재 단계(예: 조사, 해석, 설계 중)
	- 완료된 작업 목록
	- 다음 작업(우선순위)
	- 리스크/이슈
	- 요청사항(결정 포인트)

### LLM 연계 자동화 제안
- PRD·Progress 문서를 소스오브트루스로 삼아 LLM에게 주기적으로 상태 요약·회의록·리스크 리포트를 자동 생성하게 함.
- 계산 결과(예: CSV, 리포트)를 LLM이 읽어 감도 분석이나 변경 요약을 작성하도록 자동화하면 보고 시간이 단축됨.

### Confluence(Atlassian) 연동 — PRD push/pull 워크플로우
- 목적: `PRD.md`를 팀의 단일 소스오브트루스(Single Source of Truth)로 Confluence에 올리고, 팀의 실시간 협업·수정 결과를 안전하게 로컬 문서와 동기화(pull)하는 것을 표준화한다.

- 기본 원칙
	- Confluence 페이지는 공식 PRD의 "공식 버전"으로 간주하되, 로컬 `PRD.md`는 편집·검토의 준비용 복사본으로 유지한다.
	- 권한·검토 프로세스를 정의하여 무분별한 직접 변경을 막고, 누가 변경했는지와 변경 사유를 반드시 남기게 한다(페이지 버전·comment 활용).

- 권장 작업흐름(간단)
	1) 초기 작성: 로컬에서 `PRD.md`를 작성 → Confluence에 새 페이지로 push(createConfluencePage).
	2) 협업 단계: 팀원들이 Confluence에서 직접 수정(댓글·인라인 편집) → 주기적 또는 수동으로 변경사항을 로컬로 pull(getConfluencePage / fetch)하여 로컬 문서와 병합.
	3) 병합 규칙: 충돌 시에는 Confluence 변경사항을 우선 검토(변경자·버전 로그 확인) 후 로컬 PRD를 업데이트하고, 필요한 경우 Confluence에 다시 updateConfluencePage.
	4) 릴리즈/승인: 주요 변경은 '검토 → 승인' 프로세스를 거쳐 Confluence 페이지에 버전메시지를 남긴 뒤 배포(공식화).

- 자동화/툴링 제안
	- 사용 툴: Atlassian-MCP-Server API(예: `createConfluencePage`, `updateConfluencePage`, `getConfluencePage`, `searchConfluenceUsingCql`)를 사용해 스크립트화.
	- 템플릿: Confluence에 PRD 템플릿을 만들고, push 스크립트가 템플릿 기반으로 페이지를 생성하도록 통합.
	- 변경 로그 표준화: 모든 변경은 '변경 요약' 필드를 요구하고, 머지/승인 완료 시 체크리스트를 사용.

- 동기화 예외 및 운영 팁
	- 오프라인 변경(로컬에서 대규모 편집) 시에는 "Draft → Review" 절차를 거쳐 Confluence에 업데이트.
	- 자동 Pull은 소규모 변경 요약이 있을 때만 적용하고, 큰 변경은 수동 병합을 권장.
	- 정기적인 정합성 검사(예: 주 1회)로 Confluence와 로컬 PRD 간의 차이를 검토.

### 예시 스크립트(의사코드, 요약)
- Push (로컬 → Confluence)
	- if page not exists: call createConfluencePage(space, title, bodyFromPRD)
	- else: call updateConfluencePage(pageId, bodyFromPRD, versionMessage)

- Pull (Confluence → 로컬)
	- call getConfluencePage(pageId, expand=body)
	- extract body → generate PRD.md 변경요약을 만들고, 로컬과 병합(자동/수동 선택)
